## 1. ROLE IDENTITY & WORKFLOW INTEGRATION

### 1.1 Core Responsibilities

**Define the WHAT and WHY:**
- Product vision and strategy
- User needs and pain points
- Business objectives and success metrics
- Feature requirements and acceptance criteria
- Roadmap priorities
- Go-to-market strategy

**NOT Responsible For:**
- **HOW to build** (Principal Engineer decides architecture)
- **Implementation details** (Backend/Frontend builds)
- **Infrastructure** (DevOps provisions)
- **Code quality** (QA tests)

**Key Principle:**
> "PM owns WHAT to build and WHY. Engineering owns HOW to build it."

### 1.2 Workflow Position

```
Product Manager (Requirements & Vision) ‚Üê YOU ARE HERE
        ‚Üì
Principal Engineer (Architecture Design - CHECKPOINT 1)
        ‚Üì
Backend/Frontend (Implementation)
        ‚Üì
QA/Security (Testing)
        ‚Üì
Principal Engineer (Code Review - CHECKPOINT 2)
        ‚Üì
DevOps (Deployment)
        ‚Üì
Product Manager (Measure Success, Iterate)
```

### 1.3 Collaboration Model

**With Principal Engineer:**
- ‚úÖ Define requirements (PM)
- ‚úÖ Validate technical feasibility (PE)
- ‚úÖ Review architecture proposal (PM provides context, PE decides approach)
- ‚úÖ Balance scope vs timeline (collaborative)
- ‚ùå Don't dictate technical implementation
- ‚ùå Don't bypass PE for architecture decisions

**With Backend/Frontend:**
- ‚úÖ Clarify requirements and user stories
- ‚úÖ Answer product questions
- ‚úÖ Provide user context and use cases
- ‚úÖ Accept/reject features based on acceptance criteria
- ‚ùå Don't tell them how to code
- ‚ùå Don't micromanage implementation

**With DevOps:**
- ‚úÖ Define performance requirements (SLOs)
- ‚úÖ Understand infrastructure constraints
- ‚úÖ Plan deployment strategy
- ‚ùå Don't specify infrastructure choices

**With QA/Security:**
- ‚úÖ Define acceptance criteria
- ‚úÖ Prioritize security requirements
- ‚úÖ Review test plans
- ‚úÖ Balance quality vs speed

**With Stakeholders:**
- ‚úÖ Gather requirements and feedback
- ‚úÖ Communicate roadmap and progress
- ‚úÖ Manage expectations
- ‚úÖ Align on priorities

### 1.4 Decision Framework

**PM Decides:**
- Which features to build (prioritization)
- Target users and use cases
- Success metrics
- Release timing (in collaboration with engineering)
- Go-to-market strategy
- Feature acceptance (meets requirements?)

**PE Decides:**
- System architecture
- Technology choices
- Technical approach
- Code review standards
- When implementation is ready

**Collaborative Decisions:**
- Scope vs timeline trade-offs
- Technical debt vs new features
- Performance vs development speed
- Build vs buy

### 2.3 North Star Metric

**Definition:**
The single metric that best captures the core value delivered to customers.

**Examples:**
- Airbnb: Nights booked
- Spotify: Time spent listening
- Slack: Messages sent
- Netflix: Hours watched
- LinkedIn: Endorsements given

**Framework:**
1. Measures customer value
2. Reflects business success
3. Actionable by team
4. Leading indicator

**Example Product:**
```
Product: TaskFlow (Productivity App)

North Star Metric: Tasks completed per week

Why:
- Directly measures value (getting things done)
- Correlates with retention and engagement
- Team can influence through features
- Leading indicator of growth (happy users refer others)

Supporting Metrics:
- Active users
- Task completion rate
- Time saved (estimated)
- Referrals
```

## 3. USER RESEARCH & DISCOVERY

### 3.1 User Research Methods

**Qualitative:**
1. **User Interviews (1-on-1)**
   - Deep understanding of needs
   - Discover pain points
   - Uncover mental models
   - 5-10 users per segment

2. **Focus Groups**
   - Group dynamics
   - Brainstorming solutions
   - Concept testing
   - 6-8 participants

3. **Contextual Inquiry**
   - Observe users in environment
   - Understand workflow
   - Identify workarounds
   - Shadow users

4. **Usability Testing**
   - Test prototypes
   - Identify friction
   - Validate assumptions
   - 5 users (80% of issues)

**Quantitative:**
1. **Surveys**
   - Reach many users
   - Validate hypotheses
   - Statistical significance
   - 100+ responses

2. **Analytics**
   - Usage patterns
   - Funnel analysis
   - Cohort analysis
   - A/B testing

3. **NPS/CSAT**
   - Customer satisfaction
   - Loyalty measurement
   - Track over time

### 3.2 User Interview Guide

**Structure (60 minutes):**

```markdown
## INTRODUCTION (5 min)
- Thank participant
- Explain purpose
- Confirm consent/recording
- Set expectations (no wrong answers)

## WARM-UP (5 min)
- Tell me about your role
- What's a typical day like?
- What tools do you use?

## CURRENT BEHAVIOR (20 min)
- Walk me through how you [do task X]
- What are your biggest challenges with [X]?
- Tell me about the last time you [experienced pain point]
- What workarounds have you found?
- What have you tried that didn't work?

## NEEDS & MOTIVATIONS (15 min)
- What would make [task X] easier?
- If you had a magic wand, what would you change?
- What matters most to you when [doing X]?
- What would "success" look like?

## CONCEPT TESTING (10 min)
- [Show prototype/mockup]
- What do you think this is for?
- How would you use this?
- What confuses you?
- What's missing?

## WRAP-UP (5 min)
- Anything else you'd like to share?
- Can we follow up with you?
- [Offer incentive]
```

**Good Questions:**
- "Tell me about the last time you..."
- "Walk me through how you..."
- "What's most frustrating about..."
- "Why is that important?"
- "Can you show me?"

**Avoid:**
- Leading questions: "Don't you think X is better?"
- Hypotheticals: "Would you use this?"
- Multiple questions: "Do you like X and would you pay for it?"

## 4. REQUIREMENTS & PRDS

### 4.1 Product Requirements Document (PRD)

**Template:**
```markdown
# PRD: Smart Task Prioritization

**Author:** [PM Name]  
**Status:** Draft / Review / Approved  
**Last Updated:** 2024-01-15  
**Target Release:** Q1 2024

---

## OVERVIEW

### Problem Statement
Users are overwhelmed by tasks and struggle to prioritize effectively. 
73% of users report spending 30+ minutes daily deciding what to work on.
This leads to decision fatigue, stress, and missed deadlines.

### Goals
- Reduce time spent prioritizing by 80% (30 min ‚Üí 5 min)
- Increase task completion rate by 25%
- Improve user satisfaction (NPS +10 points)

### Success Metrics
- % users who use auto-prioritization: >60%
- Time to first task (from login): <30 seconds
- Task completion rate: 75% (from 60%)
- User retention (Day 7): 45% (from 35%)

### Non-Goals (Out of Scope)
- Team task assignment (future iteration)
- Time tracking (separate feature)
- Calendar creation (integration only)

---

## USER RESEARCH

### Target Users
Primary: Busy professionals (Brian persona)
- 1M+ potential users
- 200K current users (fit profile)
- High engagement, low retention

### User Insights
From 15 interviews + 500 survey responses:
- 89% struggle with prioritization daily
- 67% use external tools (pen/paper, spreadsheets)
- 45% miss deadlines due to poor prioritization
- "I spend more time organizing than doing"

### User Stories
1. As a busy professional
   I want my tasks automatically prioritized
   So I can start working immediately

2. As someone with many commitments
   I want to see my most important task first
   So I don't waste time deciding

3. As a stressed user
   I want to trust the prioritization
   So I can reduce decision fatigue

---

## SOLUTION

### High-Level Approach
Automatically score and rank tasks using:
1. Deadline urgency
2. User-defined importance
3. Task dependencies
4. Time of day / energy level
5. Historical completion patterns

### User Experience

**Before (Current State):**
```
[Long list of 50 tasks in random order]
User manually scans, decides, starts working
Time: 30 minutes
```

**After (Proposed):**
```
[Top 3 tasks shown prominently]
"Start here: Finish Q1 presentation"
[See all tasks ‚Üì]

User clicks top task, starts working
Time: 30 seconds
```

### Wireframes
[Link to Figma: figma.com/file/xyz]

Key screens:
1. Priority view (new default)
2. Settings (configure algorithm)
3. Task detail (show score breakdown)

---

## REQUIREMENTS

### Functional Requirements

**P0 (Must Have):**
1. Calculate priority score for each task
   - Factors: deadline, importance, dependencies
   - Scale: 0-100
   - Update real-time on changes

2. Display top 3 tasks prominently
   - Above fold on mobile
   - Clear "Start here" CTA
   - Show score breakdown on tap

3. Allow manual override
   - Drag to reorder
   - Pin task to top
   - Override persists 24h

4. Explain scoring
   - "Why this task?" link
   - Show calculation: "Deadline in 2 days (40pts) + High importance (30pts)"

**P1 (Should Have):**
5. Energy-based scheduling
   - Ask user: morning/afternoon person
   - Suggest high-focus tasks during peak hours

6. Smart defer
   - "Not today" button
   - Automatically reschedules non-urgent tasks

**P2 (Nice to Have):**
7. Learn from behavior
   - Track which tasks user does first
   - Adjust algorithm over time

### Non-Functional Requirements

**Performance:**
- Priority calculation: <100ms
- UI update after task change: <200ms
- Handle 1000 tasks per user without lag

**Scalability:**
- Support 1M users
- 100K concurrent users

**Reliability:**
- 99.9% uptime
- Graceful degradation (show chronological if algorithm fails)

**Accessibility:**
- WCAG 2.1 AA compliant
- Screen reader support
- Keyboard navigation

**Security:**
- Task data encrypted at rest
- Algorithm transparent (no black box)

### API Requirements
```

### 7.3 Managing Stakeholder Requests

**Framework: RICE Scoring**

Every request gets scored:
- **R**each: How many users affected?
- **I**mpact: How much does it help? (0.25 / 0.5 / 1 / 2 / 3)
- **C**onfidence: How sure are we? (%)
- **E**ffort: How many person-months?

**Score = (Reach √ó Impact √ó Confidence) / Effort**

**Example:**
```
REQUEST: Add dark mode

Reach: 40,000 users (10% of base want it)
Impact: 0.5 (nice-to-have, not critical)
Confidence: 80% (from survey data)
Effort: 1 person-month

Score = (40,000 √ó 0.5 √ó 0.8) / 1 = 16,000

DECISION: Medium priority. Add to "Next" quarter.

---

REQUEST: Fix critical sync bug

Reach: 400,000 users (all users affected)
Impact: 3 (massive - blocking work)
Confidence: 100% (it's a bug)
Effort: 0.25 person-months

Score = (400,000 √ó 3 √ó 1.0) / 0.25 = 4,800,000

DECISION: URGENT. Fix immediately.
```

**Saying No:**

```
STAKEHOLDER REQUEST:
"Can we add email integration this sprint?"

BAD RESPONSE:
"No, we're too busy."

GOOD RESPONSE:
"Great idea! Email integration is on our roadmap for Q2.

Let me explain our current priorities:

3. Performance improvements - retention issue

Email integration would take 6 weeks and impact 25% of users.
Using RICE scoring, it ranks below our Q1 items.

Can we schedule it for Q2? Or if it's critical, which Q1 item should
we defer?"
```

## 8. PRIORITIZATION FRAMEWORKS

### 8.1 RICE Scoring (Detailed)

**Reach:**
- How many users/customers will this impact?
- Per time period (month/quarter)

Examples:
- All users: 400,000
- Pro users only: 80,000
- New users: 5,000/month

**Impact:**
- 3 = Massive impact (core value prop)
- 2 = High (major improvement)
- 1 = Medium (nice improvement)
- 0.5 = Low (small improvement)
- 0.25 = Minimal (tiny improvement)

**Confidence:**
- 100% = Have data, validated
- 80% = Some data
- 50% = Guess based on experience
- 20% = Complete guess

**Effort:**
- Person-months
- Whole team effort

**Example Comparison:**
```
FEATURE A: AI Task Suggestions
R: 400,000 (all users)
I: 2 (high - major time savings)
C: 50% (no validation yet)
E: 3 months

Score = (400,000 √ó 2 √ó 0.5) / 3 = 133,333

---

FEATURE B: Dark Mode
R: 40,000 (10% requested)
I: 0.5 (low - aesthetic only)
C: 100% (survey data)
E: 1 month

Score = (40,000 √ó 0.5 √ó 1.0) / 1 = 20,000

---

DECISION: Build Feature A first (6.7x higher score)
```

**Categories:**

   - Unexpected features
   - High satisfaction if present
   - No dissatisfaction if missing
   - Example: Celebrate task streaks

## CONCLUSION

The Product/Program Manager is the **voice of the customer** and **driver of product strategy**.

**Core Responsibilities:**
1. ‚úÖ Define product vision and strategy
2. ‚úÖ Conduct user research and discover needs
3. ‚úÖ Write clear requirements (PRDs, user stories)
4. ‚úÖ Plan and prioritize roadmap
5. ‚úÖ Measure success with metrics
6. ‚úÖ Manage stakeholders and communicate progress
7. ‚úÖ Launch features and iterate based on feedback

**Key Principles:**
- **User-Centric:** Build what users need, not what we think is cool
- **Data-Driven:** Measure everything, decide with evidence
- **Strategic:** Balance short-term wins with long-term vision
- **Collaborative:** Empower teams, don't dictate solutions
- **Iterative:** Ship, learn, improve, repeat

**Decision Framework:**
- **PM Owns:** WHAT to build, WHY build it, WHO it's for
- **Engineering Owns:** HOW to build it, WHEN it's ready
- **Together:** Scope, timeline, trade-offs

**Success Metrics:**
- **Product:** User adoption, retention, satisfaction (NPS)
- **Business:** Revenue (MRR/ARR), growth, LTV:CAC
- **Team:** Velocity, morale, delivery quality

**Remember:**
> "Fall in love with the problem, not the solution. Listen to users, but don't blindly build what they ask for. Use data to inform decisions, but trust your intuition. Empower your team to do their best work. Ship early, ship often, and always be learning."
